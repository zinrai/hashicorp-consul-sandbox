# Rolling server upgrade (fixed IP, fresh provision)

How to upgrade a Consul **server** cluster across versions when the deployment fixes
server IPs and *replaces* hosts rather than upgrading them in place. The steps
themselves are generated by
[consul-server-runbook](https://github.com/zinrai/consul-server-runbook); this page is
the reasoning behind them. Map the premises onto your own cluster.

## Use case and premises

Suppose:

- A five-server Consul cluster is to move from one version to the next (the example
  here uses `consul:1.8.7` to `hashicorp/consul:1.22.7`).
- **IPs are fixed and membership does not depend on DNS**: `retry_join` lists server
  IPs, not names, so a DNS outage cannot destabilise Raft.
- Each server is **replaced, not upgraded in place**: the old host is shut down and a
  freshly provisioned new-version host takes over **its IP**. A new disk means a new
  node-id, so the old node is removed from Raft with `consul force-leave -prune`.
- Clients are upgraded after the servers (servers first, clients last); during the
  window the clients are still on the old version.

Given those premises, the operation replaces the servers **one at a time, followers
first and the leader last** (exactly one election, at the end), keeping a voting quorum
at every step.

> A same-disk in-place image swap (no prune) is a simpler smoke test, not this
> operation; the [README](../README.md) shows that variant.

## Generate the steps

In the sandbox each `consul-server-N` stands in for an old-and-staged host pair: it is
torn down and recreated at the same pinned IP with a fresh volume and the new image.
Pin the baseline in `.env` and bring the cluster up:

```
printf 'CONSUL_SERVER_IMAGE=consul:1.8.7\nCONSUL_CLIENT_IMAGE=consul:1.8.7\n' > .env
docker compose up -d
```

The leader-last procedure itself is generated by
[consul-server-runbook](https://github.com/zinrai/consul-server-runbook) against these
five servers, from its sandbox example template (see consul-server-runbook for how to
run it). The generated `runbook.md` has a preamble that starts `consul-server-tail` and
`consul-client-tail` (against a `clients.json` inventory you provide) and seeds a canary
key, then, per server in followers-first / leader-last order, the `docker compose rm` /
`force-leave -prune` / `docker volume rm` / `docker compose up` commands plus the
per-step `consul-state-dump` / `consul-state-diff`. Follow it.

(`ui = true` in `config/ui` is the portable UI setting; the `ui_config` block is 1.9+,
so `ui = true` is what boots every version.)

## Watch while you migrate

- **[consul-server-tail](https://github.com/zinrai/consul-server-tail)**: the live
  quorum view (voters, leader, index convergence), for the whole window.
- **consul-client-tail**: the old clients' local access path, the residual the servers
  cannot see; point it at a `clients.json` you write (see consul-client-tail's example).
- **[consul-state-verify](https://github.com/zinrai/consul-state-verify)**: the per-step
  `consul-state-dump` / `consul-state-diff` the runbook already emits, judging that KV /
  services / catalog survived.

## What a clean run looks like

- All five servers end on the new version with **new node-ids** at the **same IPs**;
  the leader has moved exactly once.
- `consul-state-diff` reports **PASS**: the seeded KV / services / catalog survive, and
  the replaced servers show as benign **re-registration** (same name, new id, advanced
  index), not loss or corruption.
- Client impact stays confined to **transient KV-read blips** at each replacement (a
  forward read landing on a server mid-swap); DNS service discovery and the agents' own
  checks stay green.
